# Scripts directory

Most files here are **one-off** operational tools (migrations, backfills,
admin fixes). They're committed for auditability, not because they're part
of the runtime. Running them against production Supabase is usually
destructive — read each one before invoking.

All scripts expect `.env.local` at the repo root with Supabase credentials.

## Active / reusable

| Script | Purpose |
|---|---|
| `apply-migration.ts` | Apply a single SQL migration file via direct `pg` (requires `DATABASE_URL`). |
| `setup-amenities.ts` | Create the `amenities` reference table and seed it. Idempotent. |
| `setup-storage.ts` | Provision the `hotel-images` bucket. Idempotent. |
| `update-bucket.ts` | Re-apply the hotel-images bucket policy (3MB cap, jpeg/png/webp only). |
| `stress-test.js` | k6-style load test. Used for pre-launch capacity testing. |
| `sync-hotels-from-xlsx.ts` | Reconcile hotel/room data from `data/xlsx-sync-*.json`. Writes a dry-run match report first. |
| `set-super-admin.ts` | Grant `super_admin` role to a user by email. |

## Guidelines for new scripts

1. Use TypeScript (`.ts`) and the `@supabase/supabase-js` client. Avoid `pg` unless you truly need direct SQL — pooler DNS is blocked from some networks.
2. Require `dotenv.config()` at the top.
3. Default to **dry-run**; require `--apply` to actually write.
4. Print a summary line (count of rows inspected / updated).
5. Exit non-zero on any error.
6. Add an entry to this table on merge.
